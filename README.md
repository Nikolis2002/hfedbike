# Hierarchical Federated Learning for Bike-Sharing Demand

Hour-ahead demand forecasting for the New York City Citi Bike network,
trained with a hybrid federated-learning architecture: four geographic
neighborhoods, static leaders, peer-to-peer ring reduce inside each
neighborhood, and a non-computational router between leaders.

## Quickstart

Prerequisites: Docker, and an external `my_macvlan` network on the host
(one-time setup — replace `eth0` with your active interface):

```bash
docker network create -d macvlan \
  --subnet=192.168.1.0/24 --gateway=192.168.1.1 \
  -o parent=eth0 my_macvlan
```

### Run the federation

```bash
# 1. Build the image (once, or after requirements.txt changes)
docker build -t federated-node .

# 2. Clear previous results if you want a clean run:
docker run --rm -v "$(pwd)/results/current:/r" alpine rm -f /r/*.csv

# 3. Launch 16 nodes + the coordinator
NODE_SCRIPT=v2_node.py docker compose up -d

# 4. Wait ~20 minutes; follow progress with
docker logs -f nodeX

# 5. When all nodes are in Exited(0), regenerate plots and summary stats:
python3 analysis/test_for_year.py
```

Per-node CSVs land in `results/current/<NODE>_results.csv` with columns
`week, base_mae, base_mse, freeze_mae, freeze_mse, fed_mae, fed_mse`.
`analysis/test_for_year.py` aggregates them, writes
`figures/mae_comparison.{pdf,png}`, and prints the weekly-average MAE for
each model plus pairwise Wilcoxon signed-rank p-values.

### Regenerate the baseline model (optional)

```bash
# Grid search over ~1,700 configurations (hours of CPU time)
python3 baseline/model_search/find_the_best_baseline.py

# Retrain the winning configuration on the full 2023 data
python3 baseline/model_search/best_baseline_model.py
# Writes data/2024/_model_two_layers.keras (overwrite the in-tree baseline)
```

## Headline results

On the 52 consecutive ISO weeks of 2024, across 16 subzones:

| Model                  | Avg weekly MAE (z-score units) | 95% CI          |
|------------------------|-------------------------------:|-----------------|
| Frozen baseline        | 0.648                          | [0.635, 0.661]  |
| Bias-adapted baseline  | 0.571                          | [0.561, 0.582]  |
| **Federated**          | **0.459**                      | [0.446, 0.472]  |

Federated vs. frozen: **−29.2%**. Federated vs. bias-adapted: **−19.6%**.
All three pairwise differences significant at `p < 1e-8`
(paired Wilcoxon).

## Data notes

- Raw 2023 CSVs in `data/2023/` were downloaded from
  <https://citibikenyc.com/system-data>; weather from OpenWeatherMap.
- Per-subzone CSVs in `data/2024/entire_year/` are the output of
  `baseline/data_preparation/make_neighbs_2024/make_the_final_csvs.py`
  (station → k-means subzone mapping, hourly aggregation).
- The frozen baseline keras model lives at
  `data/2024/_model_two_layers.keras`; both `federated/p2p_node/model.py`
  and the docker-compose mount expect it there.

## Branches

The repo is organized around a stable trunk with two lines of branches:
deployment variants (different ways to run the same production
protocol) and research variants (shadow-metric ablations that log
additional signals alongside the deployed run).

### Main trunk

- **`main`** — production code. The `v2_node.py` federated pipeline, the
  full 16-node `docker-compose.yml` on macvlan, and the analysis
  scripts used to produce the paper's Table I. This is the branch the
  paper's numbers come from.

### Deployment variants

- **`hybrid`** — Raspberry-Pi-in-the-loop deployment. Adds
  `pi_node/` (self-contained single-service compose for the Pi, macvlan
  setup script, subset of data needed for nodeA, step-by-step README)
  and `docker-compose.hybrid-host.yml` (the 15 host containers, nodeA
  omitted because the Pi fills that slot). The MDM'26 demo submission
  references this branch. Also adds
  `analysis/weekly_improvement_bars.py` for the demo's per-week
  %-improvement figure.

### Research variants (shadow-metric ablations)

These branches modify `v3_node.py` to log one or more additional metrics
*during the same 52-week run*, without changing the deployed
algorithm. Each adds columns to the per-node CSV and includes analysis
scripts to interpret them.

- **`region3-option-a`** — corrects the Region-3 failure framing using
  only the data the production run already produced. Adds
  `analysis/region3_scatter.py` (Fed/Bias improvement vs. per-subzone
  mean demand) and the scatter figure.

- **`region3-option-c1`** — builds on `region3-option-a` and adds two
  shadow metrics to `v3_node.py`:
  *Fed-intra* (neighborhood mean weights from Step 3, skipping the
  cross-region FedAvg of Step 5) and *Fed-scale* (per-leader
  scale-aware blend driven by the Welford σ the coordinator already
  broadcasts). One extra forward pass per hour per node; same 52-week
  run. See `analysis/compare_intra_cross.py` and
  `analysis/option_b_comparison.py` for the interpretation scripts.

### Snapshots and historical

- **`restructure/cleanup`** — a pointer to the commit that introduced
  the current `data/ baseline/ federated/ results/` layout. Identical
  tip to `main`; kept as a stable reference.

- **`fix/input-normalization`** — the two correctness fixes (input
  Z-score consistency and exact-mean ring reduce) in isolation. Used
  to produce the paper's before/after comparison.

- **`demo/pi-single-neighborhood`** — earlier exploration of running a
  full 4-node neighborhood (plus coordinator) entirely on a single Pi.
  Superseded by `hybrid`, kept for reference.

### Picking a branch

| Goal                                              | Branch                  |
|---------------------------------------------------|-------------------------|
| Reproduce Table I                                 | `main`                  |
| Run nodeA on a Pi alongside 15 host containers    | `hybrid`                |
| See the regime scatter                            | `region3-option-a`      |
| Also log Fed-intra + Fed-scale during the run     | `region3-option-c1`     |
| Read the clean pre-experiment code                | `main` or `restructure/cleanup` |
